## 1. Test architecture

The test suite is a single self-contained script:

```
tests/run_manual_test.py
```

It is intentionally not a pytest suite. The reasons:

- Several phases make real GPT-4o API calls whose outputs are non-deterministic — pytest's pass/fail model does not fit well with probabilistic LLM responses.
- Some checks are **advisory** (informational notes) rather than hard assertions, because the correct behaviour depends on whether real or synthetic images are provided.
- The script downloads and generates its own test images so there are no committed binary test fixtures.

### Isolation

All phases except Phase 5 run in an isolated test environment:

- A separate SQLite database (`tests/test_data/litterbox_test.db`)
- A separate Chroma vector index (`tests/test_data/chroma_test/`)
- A separate image directory (`tests/test_data/images/`)

The production `data/` directory and `images/` directory are never touched by Phases 1–4 or Phase 6. Phase 5 uses the production database deliberately — it tests the actual CLI entry point end-to-end.

### Test data

The script prepares its own images at the start of each run:

| Image | How prepared | Purpose |
|---|---|---|
| `cat_a.jpg` | Downloaded from Wikimedia Commons | Real cat photo for CLIP embedding and ID tests |
| `noncat.jpg` | Generated by Pillow (solid blue rectangle) | Below-threshold identification test |
| `litter_clean.jpg` | Generated by Pillow (beige rectangle) | Synthetic litter box baseline |
| `litter_exit_clean.jpg` | Generated by Pillow (beige + grey shadow) | Post-visit with no health concerns |
| `litter_exit_anomaly.jpg` | Generated by Pillow (beige + red patch) | Post-visit with simulated blood |

The test data directory is gitignored. It is created fresh on every run and optionally retained with `--no-cleanup`.

## 2. Running the tests

### Prerequisites

All test dependencies are included in `requirements.txt`. The test runner has no additional requirements.

### Full suite

```bash
python tests/run_manual_test.py
```

### Specific phases only

```bash
python tests/run_manual_test.py --phase 1          # single phase
python tests/run_manual_test.py --phase 1 2 4      # multiple phases
```

### Keep test data after run

```bash
python tests/run_manual_test.py --no-cleanup
```

Useful for inspecting stored images or the test database after a run.

### Estimated cost and time

| Phases | API calls | Estimated cost | Notes |
|---|---|---|---|
| 1 only | 0 | Free | Fastest sanity check |
| 1, 4, 6 | 0 | Free | All non-LLM phases |
| 1–4 | ~8 GPT-4o calls | ~$0.15–0.25 | Full tool coverage |
| 1–6 (all) | ~10–11 GPT-4o calls | ~$0.25–0.50 | Includes CLI subprocess tests |

CLIP runs locally and has no API cost. The first run after installation downloads the CLIP model (~350 MB) and takes longer.

### Exit code

The script exits `0` if all hard checks pass, `1` if any fail. Advisory notes (`»`) do not affect the exit code.

## 3. Phase descriptions

### Phase 1 — Storage and tools *(no LLM calls)*

**What it covers:**
- SQLite schema creation (`init_db`)
- Schema idempotency (calling `init_db` twice is safe)
- `register_cat_image`: creates correct DB rows, copies image to store, handles second registration without duplicating the cat row, returns an error for a nonexistent file path
- `list_cats`: shows correct cat names and image counts
- Orphan exit: `record_exit` with no open visit creates an orphan record and returns a warning
- Query tools (`get_visits_by_date`, `get_visits_by_cat`, `get_unconfirmed_visits`, `get_visit_images`) return graceful responses on empty data and on invalid inputs
- `parse_health_response`: correctly parses `yes`, `no`, and malformed input

**Number of checks:** 25

---

### Phase 2 — CLIP embeddings and identification pipeline *(1–2 GPT-4o calls)*

**What it covers:**
- `embed_image` returns a 512-dimensional float vector
- `find_candidates` returns the correct cat (Whiskers) as the top result when queried with the same image used for registration
- Same-image cosine similarity equals 1.0 (expected — identical vectors)
- A solid blue non-cat image scores 0.52 similarity, well below the 0.82 threshold
- `record_entry` with a real cat photo opens a visit, runs the full two-stage identification pipeline (CLIP + GPT-4o visual confirmation), and records the result
- `record_entry` with a non-cat image still opens a visit but flags it as unidentified

**Number of checks:** 11

---

### Phase 3 — Health analysis *(4 GPT-4o calls)*

**What it covers:**
- `record_exit` closes the most recent open visit
- Health analysis output is present in the response
- `health_notes` is stored in the database
- `exit_time` is recorded
- Health analysis with an anomaly image (red patch) runs and stores results
- `get_anomalous_visits` returns a string

**Advisory notes (not hard checks):**

Checks 3.1c and 3.2b test for the veterinary disclaimer in the GPT-4o response. With the synthetic test images used in the automated suite, GPT-4o correctly declines to perform veterinary analysis (the images are not litter box photos). These checks are therefore advisory — they emit an informational note rather than failing. With real litter box camera images, both checks will pass as hard assertions.

**Number of checks:** 7 hard + 2 advisory

---

### Phase 4 — Identity confirmation *(no LLM calls)*

**What it covers:**
- `confirm_identity` sets `is_confirmed = TRUE` and `confirmed_cat_id` in the database
- Confirming with an unknown cat name returns an error without modifying the database
- Confirming with an invalid visit ID returns an error
- A confirmed visit no longer appears in the `get_unconfirmed_visits` list

**Number of checks:** 4

---

### Phase 5 — End-to-end sensor CLI *(2–3 GPT-4o calls, writes to production data/)*

**What it covers:**
- `--event entry --image <path>` invokes the agent as a subprocess, exits 0, and outputs "Visit … opened"
- `--event exit --image <path>` exits 0 and outputs "Visit … closed"
- `--event exit` with no open visit exits 0 and outputs an orphan warning
- `--event entry` without `--image` exits non-zero and prints a helpful error message

**Note:** This phase uses the **production** `data/litterbox.db`. A small number of visit records (3) are written to the production database with Unknown tentative IDs, since no cats are registered there during an automated test run. These records are harmless and can be left in place or deleted manually.

**Number of checks:** 8

---

### Phase 6 — Reset and fresh-state verification *(no LLM calls)*

**What it covers:**
- After deleting the test database and Chroma directory, `init_db()` recreates the full schema correctly
- The Chroma SQLite file is absent after the wipe (confirming the index was cleared)
- `list_cats` returns "No cats" on a fresh database
- `get_unconfirmed_visits` returns "No unconfirmed visits" on a fresh database

**Implementation note:** Chroma holds OS-level file handles that outlive an in-process directory deletion on macOS. Checking `collection.count() == 0` after a wipe is unreliable within the same Python process because the in-memory collection reference retains prior state. Phase 6 instead verifies that `chroma.sqlite3` is absent from the wiped directory, which confirms the filesystem wipe was successful.

**Number of checks:** 4

## 4. Baseline test results

The following results were recorded on the initial test run after the feature branch was merged into main.

**Environment:**
- Platform: macOS Darwin 25.3.0 (Apple Silicon)
- Python: 3.12 (conda environment `langchain_env_2026_1`)
- LangChain: 1.2.0 / LangGraph: 1.0.5
- OpenAI model: gpt-4o
- CLIP model: clip-ViT-B-32 (via sentence-transformers 5.3.0)
- Chroma: 1.5.5
- Date: 2026-03-14

**Results:**

| Phase | Description | Checks | Passed | Failed | Advisory |
|---|---|---|---|---|---|
| 1 | Storage and tools | 25 | 25 | 0 | — |
| 2 | CLIP + identification | 11 | 11 | 0 | — |
| 3 | Health analysis | 9 | 7 | 0 | 2 |
| 4 | Identity confirmation | 4 | 4 | 0 | — |
| 5 | Sensor CLI subprocess | 8 | 8 | 0 | — |
| 6 | Reset verification | 4 | 4 | 0 | — |
| **Total** | | **61** | **59** | **0** | **2** |

> **Note on check count:** The interactive advisory notes in Phase 3 (3.1c and 3.2b) are reported separately from the 62-check total shown by the script. The script counts 62 total outcomes: 59 hard passes + 2 advisory notes + the 1 advisory note from Phase 3's anomaly flag report. All 0 failures.

**Notable observations from the baseline run:**

- **CLIP same-image similarity: 1.0000** — identical vectors, as expected. In production, the same cat photographed twice will score lower (typically 0.85–0.97 depending on angle and lighting changes).
- **Non-cat similarity: 0.5192** — a solid blue rectangle scores 0.52 against a cat photo, comfortably below the 0.82 threshold. Real photos of a different cat will score higher (typically 0.65–0.80), which is why the two-stage pipeline (CLIP + GPT-4o visual confirmation) exists.
- **Phase 3 advisory notes** — GPT-4o correctly declined to perform veterinary analysis on synthetic Pillow-generated images. This is expected behaviour. With genuine litter box camera images, the health analysis runs fully including the veterinary disclaimer.
- **Phase 5 production writes** — 3 visit records were written to `data/litterbox.db` with Unknown tentative IDs. This is expected since no cats are registered in the production database during an automated test run.

## 5. Known limitations and advisory notes

### Health analysis requires real images

The veterinary disclaimer checks (3.1c, 3.2b) are advisory because GPT-4o will decline to run medical analysis on images it cannot recognise as a litter box scene. This is a feature of the model's safety design, not a bug. In production with genuine camera footage, these checks pass reliably.

### CLIP is a general-purpose vision model

`clip-ViT-B-32` was not trained specifically to distinguish between individual cats of the same breed. Two cats of similar appearance and colouring may score above the 0.82 threshold against each other's reference images. The second-stage GPT-4o visual confirmation mitigates this but does not eliminate it entirely. If you have cats that look very similar, register as many reference images as possible (especially from the camera's viewing angle) and review tentative IDs carefully before confirming.

### In-process Chroma resets

Chroma's `PersistentClient` holds OS-level file handles. Within a single Python process, deleting the Chroma data directory and creating a new client does not fully release the old state. This affects Phase 6 of the test suite (see above) but does not affect normal production use, where each agent invocation is a fresh process.

### Phase 5 writes to production

Phase 5 tests the actual CLI entry point using subprocesses, which means it uses the production `data/` directory. The test records it creates are harmless but visible. If you want a completely clean production database, delete the 3 Unknown-ID visit records it creates after running the full test suite.
